# SQLを書く前の思考整理ガイド

SQLを書き始める前に、何をどう整理しておくとよいかをまとめたガイドです。  
正確さよりもイメージしやすさを優先して書いています。例外や解釈の違いはありますが、ざっくりとした考え方を掴む参考にしてください。

---

## 1. 用語の説明

### ディメンション（類語：粒度・分析の単位）
- **データを分析する切り口**。何ごとにデータを集計したいか。
- 例：日ごと、月ごと、年ごと、ユーザーごと、広告グループごと、キャンペーンごと、ランディングページごと、システム全体で、など

### メジャー（類語：指標・メトリクス）
- **出したい数値**。ディメンションで区切った結果、その単位の何を見たいか。
- 例：セッション数、売上、インプレッション数、クリック数、ユーザー数、ランキング順位、など

### rawデータ（生データ）
- **集計元のデータ**。絶対的な定義があるわけではなく、今からしたい集計に対する元データを指すことが多い。

### 従属（関数従属）
- あるキーが決まったら他の属性が決まる関係。
- 例：コースタイトルはコースIDに従属している。生徒名は生徒IDに従属している。
- 概念さえ理解できれば、用語を覚えなくても問題ない。

---

## 2. SQLを書く前の思考整理の手順

### 手順1：集約したメジャーを出すかどうかを決める

まず「何かを集計（集約）したメジャーを出したいか、出さなくてよいか」を判断する。

| パターン | 例 |
|---------|-----|
| **集約したメジャーを出す** | 広告グループごとのセッション数、日×キャンペーンごとのインプレッション数・クリック数・コンバージョン率 |
| **集約したメジャーは不要** | 直近3日以内に訪問のあったユーザー一覧、施策中のキャンペーン一覧 |

- **集約したメジャーが不要な場合**  
  → ディメンションも存在しないので、`GROUP BY` やウィンドウ関数の `PARTITION BY` は使わない。  
  `SELECT`, `FROM`, `JOIN`, `WHERE`, `ORDER BY`, `LIMIT` などを組み合わせて書く。

- **集約したメジャーを出す場合**  
  → 次の手順2以降に進む。

---

### 手順2：ディメンション・メジャー・フィルター条件を定義する

集約したメジャーを出す場合、以下を決める。

- **ディメンション**：何ごとに集計するか
- **メジャー**：その単位で何の数値を見るか
- **フィルター条件**：どの範囲・条件のデータを使うか（可能であればここで決める）

#### 例

| 例 | ディメンション | メジャー | フィルター条件 |
|----|----------------|----------|----------------|
| 直近1ヵ月のキャンペーンごとのインプレッション数とクリック数 | キャンペーン | インプレッション数、クリック数 | 直近1ヵ月 |
| 直近1ヵ月のキャンペーン×日ごとのインプレッション数とクリック数 | キャンペーン×日 | インプレッション数、クリック数 | 直近1ヵ月 |
| 今月の広告流入の売上合計のうち、キャンペーンごと・日ごとの割合 | 全体、キャンペーン、日 | 売上合計（全体）、売上合計（キャンペーンごと）、売上合計（日ごと）、それぞれの割合 | 今月、広告流入 |

---

### 手順3：出力する単位・rawデータ・出力カラムを整理する

SQLを書く前に、以下を整理しておくと迷いにくくなる。

| 整理項目 | 内容 |
|----------|------|
| **出力する単位** | 最終的に何を1行として出すか。ディメンションが1つならほぼそれと一致するが、複数ある場合やウィンドウ関数を使う場合は最初に決めておくとよい。 |
| **rawデータ** | 各メジャーを計算するために必要な元データ。最初のうちはメジャーごとに整理するとよい。複数メジャーで同じになることも多い。 |
| **出力するカラム一覧** | 各カラムが「ディメンション」「メジャー」「ディメンションに従属した属性」のどれに当たるかを分類できるとよい。 |

**ポイント**：従属した属性（例：コースIDに紐づくコースタイトル）は、最後に取得すればよい。無理に `GROUP BY` に入れなくてよい。

## 3. GROUP BY とウィンドウ関数の使い分け

### 大原則

| 方法 | 出力結果の単位 |
|------|----------------|
| **GROUP BY** | `GROUP BY` で指定した単位に**集約される**。行数はその単位の数になる。 |
| **ウィンドウ関数** | `FROM` / `JOIN` で取得した単位が**そのまま残る**。集約されない。 |

### 使い分けの基準

- **最終の出力結果をディメンションの単位に集約したい** → `GROUP BY` を使う
- **最終の出力結果をディメンションの単位に集約したくない**（例：集計値に加えて、その内訳のIDなども見たい） → ウィンドウ関数を使う

### なぜ複数ディメンションだとウィンドウ関数が便利なことがあるか

- `GROUP BY` を使うと、指定したディメンションの単位に結果が集約される。
- ディメンションが複数ある場合、`GROUP BY` だとディメンションの数だけ別々のサブクエリ（`WITH` 句など）を作る必要が出ることがある。
- ウィンドウ関数は集約しないので、`PARTITION BY` でディメンションを変えるだけで、一度に複数のディメンションに対するメジャーを出せる。

---

## 4. 定義に基づいてSQLを書く

### GROUP BY を使う場合

- **ディメンションの追加**：`SELECT` 句と `GROUP BY` 句に書く。ディメンションが「全体」のときは `GROUP BY` への追加は不要。
- **メジャーの追加**：`COUNT()`, `SUM()`, `AVG()` などの集約関数を使って `SELECT` 句に書く。
- **元データ**：`FROM`, `JOIN` で取得し、`WHERE` で絞る。
- **ディメンションが複数ある場合**：1つの単位に集約されるため、それぞれのディメンション単位ごとに `GROUP BY` するサブクエリを作る方法がおすすめ。

### ウィンドウ関数を使う場合

- **ディメンションの追加**：`SELECT` 句に書く。ウィンドウ関数内の `PARTITION BY` にも書く。ディメンションが「全体」のときは `PARTITION BY` への追加は不要。
- **メジャーの追加**：ウィンドウ関数を使って `SELECT` 句に書く。`PARTITION BY` にはそのメジャーに対応するディメンションを書く。
- **元データ**：`FROM`, `JOIN` で取得し、`WHERE` で絞る。
- **ディメンションが複数ある場合**：`PARTITION BY` で指定するディメンションを変えるだけでよいので、サブクエリの数が少なくて済むことが多い。

---

## 5. まとめ：SQLを書く前のチェックリスト

1. **集約したメジャーを出すか** → 不要なら `GROUP BY` / ウィンドウ関数は使わない
2. **ディメンション・メジャー・フィルター条件**を定義する
3. **出力する単位・rawデータ・出力カラム一覧**を整理する
4. **出力を集約するかどうか**で `GROUP BY` かウィンドウ関数かを決める
5. 定義に沿ってSQLを書く

この順番で整理しておくと、初学者でも「何を出したいか」がはっきりし、SQLの構造を組み立てやすくなります。
