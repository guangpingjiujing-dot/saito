# データモデルを理解する

- テーブル同士のつながりを図や言葉でイメージできると、SQL の `JOIN` が組み立てやすくなる。
- **正確さよりイメージのしやすさ**。表記や厳密な定義は教材・現場で違う。考え方のたたき台として使う。
- [SQL実装の言語化.md](./SQL実装の言語化.md)（SQLを書く前の思考整理）とあわせると、流れがつながる。

## この資料と [SQL実装の言語化.md](./SQL実装の言語化.md) のつながり

- [SQL実装の言語化.md](./SQL実装の言語化.md) の **手順3：出力する単位・rawデータ・出力カラムを整理する** では、集計・一覧のために **どのテーブルを「元データ」とするか** を整理する。
- データモデル（テーブル関係）が頭に入っていると、次が扱いやすくなる。
  - **どのテーブルが rawデータの中心か**（粒度が細かい側・事実が起きた側）
  - **必要なカラムのために、経由すべき `JOIN` の順**（まちがった経路で結合していないか）
  - **同じ意味のデータを二重に拾っていないか**（多対多のまま結合して行が増えすぎないか）
- **rawデータを選んでくる工程**を、感覚だけでなく「図とルール」で補強する、という位置づけ。

## エンティティとは何か

- **エンティティ**＝業務や世界の中で区切って扱いたい **「もの」「出来事」** を、データとして表した単位。

| イメージ | 例 |
|----------|-----|
| モノ・人・組織 | 生徒、コース、商品、顧客 |
| 出来事・記録 | 登録（enrollment）、受講レッスン（lesson）、注文 |

- 1エンティティ ≒ **1テーブル**、の対応が多い（正規化の細部はここではざっくり）。
- 「このテーブルの1行は、現実の何1件か？」を意識すると、エンティティの切り方をつかみやすい。

---

## 多対多は避けて、連関エンティティを挟む

- **多対多**＝「Aの1件がBの複数」と「Bの1件がAの複数」の両方が成り立つ関係。
  - 例：1人の生徒が複数コースに入り、1コースにも複数の生徒がいる、など。
- リレーショナルDBで多対多を **テーブル2枚だけ** で表すと、`JOIN` がわかりにくく、誤結合で行が増えることがある。
- **多対多は避ける**（設計の段階で分解する）のが一般的。

**やり方**

- 両方と **多対1** になる **連関エンティティ**（中間テーブル・関連テーブル・ジャンクション）をはさむ。
- 例：`students` と `courses` が多対多 → `enrollments`（誰がいつどのコースに登録したか）を置く。
  - `student` → `enrollment` は1対多
  - `course` → `enrollment` は1対多
- SQL では「いきなり `students` と `courses`」ではなく **`enrollments` 経由** を考えると迷いにくい。

## ER図を描くステップ

- ステップを分けると、最初から全部を完璧に描こうとしなくてよい。

### Step 1：矢印で書けるようになる

- **どちらが「依存の先」か／どちらが粒度が細かいか** を、**矢印**で表す。
- 鳥脚や多重度の記号は、この段階では必須にしないでもよい。

### Step 2：多重度を鳥脚で表現できるようになる

- 関係が固まったら **1対1／1対多／多対多** を **鳥脚記法** どで表す練習をする。
- あわせて、**片側から見た件数の下限・上限**（例：**0以上**か**ちょうど1**か、**1以上**か）を線や記号で示せるようにする。
- 多対多が出たら **連関エンティティを足して**、**1対多の組み合わせ** に直す、までをセットにする。

---

## 矢印のルール

### 1対多の「多い側」に矢印を向ける

- **1対多**なら、**多い側（同じキーで複数行になりうる側）** に矢印の先がくる、と覚えるとよい。

**例：`students` と `enrollments`**

- 結合キーは `student_id`。
- `students` 側は同じ `student_id` は1件、`enrollments` 側は同じ生徒が複数行になりうる。
- 同じ生徒が複数コースに登録できる → **矢印は `enrollments` 側（多い側）を指す**、というイメージ。

### 矢印は「依存」「出来上がりの順」を表すことがほとんど

- **例：** `enrollment`（登録）がなければ `lesson`（レッスン）は立ち上がらない、など。
- ある `lesson` が存在するなら対応する `enrollment` が必ず存在する、という関係も、矢印の向きの判断材料になる。

### 間接依存だけなら、その間に矢印を引かない

- **例：** `reviews` が `students` にだけつながっているように見えても、実際は「動画提出」など **別エンティティを経由** して初めて `review` がありうる。
- 生徒がいるだけでは review にはならない → `students` と `reviews` の間にむやみに矢印を引かず、**直接の前提の関係**にだけ矢印を引く。

### SQL の `JOIN` は、矢印の先から広げるのが鉄則

- 矢印の先のテーブルは、だいたい **粒度が細かい側**。
- `FROM` の起点にしやすい。`JOIN` で親や参照元を足すと読みやすいことが多い。

| 結合の例 | 起点のイメージ |
|----------|----------------|
| `enrollments` と `students` | まず `enrollments` → `students` |
| `lessons` と `enrollments` と `students` | まず `lessons` → 次に `enrollments` → 必要なら `students` |

## まとめ

- **エンティティ**＝業務上の「もの・出来事」をデータの単位として切ったもの（多くの場合テーブルに対応）。
- **多対多**は設計も SQL も扱いにくい → **連関エンティティ**で **1対多×2** にばらす。
- **ER図**は **Step1：矢印** → **Step2：鳥脚で多重度** の順で慣れるとよい。
- 矢印の向きと **`JOIN` の起点**をそろえると、[SQL実装の言語化.md](./SQL実装の言語化.md) の **rawデータ選定・結合の整理**につながる。
